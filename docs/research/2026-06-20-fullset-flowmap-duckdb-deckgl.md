# Research Brief — Full-Dataset OD/Flow Viz Architecture (Flowmap.gl / DuckDB-WASM / deck.gl)

Date: 2026-06-20
Scope: Next-gen architecture to move the Catalonia MITMA v2 OD-flow viz from a tiny precomputed `arcs.json` sample to an interactively sliceable view over the full hourly × daily × season × demographic OD cube — under the project's hard constraints: **keyless, no build step (CDN `<script>` / import-map only, no bundler, no React), fully static on GitHub Pages, deck.gl 9.3.2 + MapLibre 4.7.1 + h3-js 4.1.0 already in use.**

## Known (internal)

- **Render side is already decided and binding (OB1).** Standardize on the deck.gl/vis.gl family as the rendering primitives; MapLibre is a keyless basemap host only, not the geometry/flow renderer (OB1 `f03b8f7e`). Native-MapLibre custom-WebGL arcs were "a constant fight" and are parked. Any new OD-flow viz is a deck.gl layer.
- **The live arc renderer is a deck.gl `ArcLayer`** (`greatCircle:true`, `getHeight:0.4`), width = `log10(flow+1)` mapped linearly into ~1.5–8px from a `logMin/logMax` scanned on load, with `updateTriggers:{getWidth:[logMin,logMax]}` (`docs/app/geobrowser-map.js` lines ~758–814; OB1 `7b2b1b72`). **Hard lesson:** presentation was silently coupled to data magnitude — when full-scale flows regenerated `arcs.json`, the width formula saturated. Width must stay **data-driven (domain scanned at load)** and guarded by a visual-regression gate.
- **Blending is a hard constraint.** At the Barcelona hub ~5000 corridors converge; additive blending blows out to ~47% white pixels. The renderer MUST use blowout-safe alpha-over (premultiplied `ONE, 1-SRC_ALPHA`); density comes from more corridors + flow-driven width + soft glow, NOT additive accumulation (OB1 `222645be`, `2d9b083a`).
- **Data side is GREENFIELD.** OB1 has **no** prior thought on DuckDB-WASM, Parquet-over-HTTP-range, Arrow, or flowmap aggregation. Client-side query architecture is the open design question.
- **Current loading is whole-file eager/lazy JSON.** Eager: `hexes.json` + `manifest.json`. Lazy-on-first-use: `arcs.json`, `pois.json`, `rhythm.json`, `seasons.json`. No Arrow/Parquet/DuckDB, no HTTP range usage, no binary format anywhere in the client today.
- **Measured sizes on disk (today, verified):** `hexes.json` = **69.3 MB** (not the 54 MB the brief cites — already over GitHub's 50 MB soft warning); `seasons.json` 20.2 MB; `rhythm.json` 9.7 MB; `arcs.json` **580 KB / 5000 records** (grown from the 250-record sample); `pois.json` 23 KB; `manifest.json` 5.7 KB. Full OD source: `data/silver/od_silver` = **3.6 GB**, 89 daily Parquet partitions, ~4.0M rows/day; schema `periodo(hour)/origen/destino/distancia/actividad_*/residencia/renta/edad/sexo/viajes/...`. Small gold intermediates exist (`h3_mitma_features.parquet` 7.1 MB, `seasonal_long` 4.4 MB).
- **Hosting constraint mapped (OB1 `aaa80677`).** GitHub Pages serves HTTP **range** requests (probed `206` + `accept-ranges: bytes`). GitHub **Releases do NOT send CORS** → big OD data must live **same-origin under `docs/`**, never as a Release asset.
- **`hexes.json` is a firm boundary.** It must stay byte-identical and is already 69 MB. The OD-flow store MUST be an independent sibling — never merged into `hexes.json`, which cannot grow.

## New (external, cited)

- **Flowmap.gl is now under the official vis.gl org and deck.gl-9-aligned.** `@flowmap.gl/layers@9.4.0` (latest, 2026-06-16) is a real deck.gl `CompositeLayer`; peerDeps `@deck.gl/core ^9.0.0` + `@luma.gl/* ^9.0.0`; Apache-2.0; ships zoom-based clustering, adaptive scales, top-N filtering, viewport-only rendering, animated particles, curved arrows, typed picking — exactly the OD feature set. https://github.com/visgl/flowmap.gl • https://github.com/visgl/flowmap.gl/blob/main/examples/vite-app/src/app.js (vanilla non-React Deck+MapLibre template) • https://www.npmjs.com/package/@flowmap.gl/layers
- **BUT it is pure ESM, no UMD.** It cannot augment the project's UMD `deck` global; mixing UMD deck.gl with an ESM deck.gl/luma for flowmap creates two incompatible instances and breaks custom shader layers. Verified `esm.sh` resolves `@flowmap.gl/layers@9.4.0`, `@flowmap.gl/data@9.4.0`, `deck.gl@9.3.2` (HTTP 200) and externalizes deck/luma peers — so an import-map page works build-free, but it is a **separate ESM context** beside the existing UMD map. https://esm.sh/@flowmap.gl/layers@9.4.0
- **DuckDB-WASM works keyless/static but single-threaded only on Pages.** httpfs range-scans remote Parquet with column pruning + predicate pushdown over `206` responses. Multi-threaded (`coi`) build needs `SharedArrayBuffer` → COOP/COEP headers, which **GitHub Pages cannot set** ("No ETA", GitHub community #13309). Default `eh` bundle is single-threaded and needs no headers. Avoid the `coi-serviceworker` hack (forces reload, fragile). ~30 MB WASM (≈6–11 MB gzip), lazy-loadable. https://duckdb.org/docs/current/clients/wasm/overview • https://github.com/orgs/community/discussions/13309
- **Range-pushdown caveat:** DuckDB-WASM has historically fallen back to FULL file reads on Pages/raw URLs (duckdb-wasm #1366/#1367, discussion #407). Prefer **your own explicit `Range` fetch against a precomputed offset index** over trusting Parquet range-pushdown on Pages. https://github.com/duckdb/duckdb-wasm/discussions/407
- **`@geoarrow/deck.gl-layers` is OUT** — fails from a CDN global without a bundler ("Cannot read properties of undefined (reading isPointVector)", issue #109; maintainer says "use ESM with a build step"). https://github.com/geoarrow/deck.gl-layers/issues/109
- **The build-free zero-copy path that DOES work** is deck.gl's native **binary `data.attributes`**: pass `data={length:N, attributes:{getSourcePosition:{value:Float32Array,size:3}, getTargetPosition:{value:Float32Array,size:3}, getWidth:{value:Float32Array,size:1}}}` straight to a stock `deck.ArcLayer`. TypedArrays upload directly to GPU, no per-row accessors. `apache-arrow` loads via `esm.sh` (`tableFromIPC`) so Arrow column buffers feed attributes with no copy. **deck.gl 9 defaults positions to `size:3`** — getting size wrong silently renders at 0,0. https://deck.gl/docs/developer-guide/performance • https://github.com/visgl/deck.gl/discussions/8984
- **`DataFilterExtension` is THE lever primitive** — GPU show/hide with zero re-upload for threshold/hour/day-type. Limits: 1–4 numeric + 1–4 category dims; `countItems`+`onFilteredItemsChange` gives a live "N of M corridors" count. Use a fixed time origin (not raw epoch ms) to avoid 32-bit float drift. Ships in `@deck.gl/extensions` UMD (CDN-loadable). https://deck.gl/docs/api-reference/extensions/data-filter-extension
- **PMTiles-over-range on Pages is broken in Firefox** (16KB-chunk cache reuse → out-of-range; protomaps #584/#582, unresolved mid-2025). If tiling is ever needed, use a pre-sliced static `{z}/{x}/{y}` MVT/GeoJSON tree (plain file fetches, Pages-safe). https://github.com/protomaps/PMTiles/issues/584

## Actionable (≤4 named next steps)

1. **Build the OD viz as a separate ESM applet (`docs/explore_flows.html`), not an injection into the live UMD map.** One `<script type=importmap>` pinning `@deck.gl/core`, `@deck.gl/layers`, `@deck.gl/extensions`, `@luma.gl/*`, `maplibre-gl@4.7`, `@flowmap.gl/data@9.4.0`, `@flowmap.gl/layers@9.4.0` to `esm.sh` on ONE pinned 9.x line. **Spike a 1-file dual-instance check FIRST** (symptom = shader/context errors) before wiring real data — that is the only realistic failure mode.
2. **Re-shape the export pipeline (`scripts/export_mitma_layers.py`) from "queries" to "levers": ship per-(season × day-type) OD slices**, each a top-N-capped arc array (target ≤10k–50k arcs/slice) carrying numeric filter fields on the record (`flow`, `flow_by_hour[24]`, `rank`), plus a tiny manifest mapping `sliceKey → URL` (or `→ [byteOffset,length]` into one concatenated binary). Keep these as new lazy siblings — **do not touch `hexes.json`.**
3. **Wire levers via `DataFilterExtension` on a stock `ArcLayer` with binary `data.attributes`** (volume threshold = `filterRange` on `log10(flow+1)`; hour scrub = animate `filterRange` over `flow_by_hour`); reserve **data-prop swaps (debounced ~200ms)** for structural changes only (season/day-type/zoning). Keep the data-driven width domain + add a **visual-regression gate** (re-asserts the OB1 magnitude-coupling + blowout fixes).
4. **Add DuckDB-WASM (single-threaded `eh`, lazy-loaded) only as an optional "explore" power-user mode** for ad-hoc origin/destination/hour pivots — feeding Arrow results into the same binary `data.attributes`. Do NOT make it the default engine; precomputed lever-slices win on payload and load time. Defer semantic-zoom h3 parent rollups and any MVT tiling as escape hatches, not v1.
